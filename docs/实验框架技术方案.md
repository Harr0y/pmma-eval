# 实验框架技术方案

> **更新日期**：2026-04-18
> **版本历史**：v1 LangGraph → v2 Claude API Python 自研 → v3 基于 disclaude + TypeScript → v3.1 确定多Agent调度架构 → v3.2 模型统一切换为 GLM-5-Turbo（所有实验统一按实际 Token 挂牌价计费，Robustness 保留 DeepSeek V3）

---

## 一、设计原则

1. **方法论与框架解耦**：Evolutionary PM（进化式项目管理）是项目管理方法，不是技术框架的功能。实验框架只是载体。Evolutionary PM 的核心策略是将项目管理重新定义为进化过程——通过变异（多个独立实现）→ 选择（TDD 测试筛选）→ 保留（提取优秀性状遗传给下一代），逐步逼近最优解。
2. **Prompt 驱动的方法差异**：五种 PM 方法的区别通过 Skill（SKILL.md）和 Agent 定义（.claude/agents/*.md）注入，不依赖代码层面的协调逻辑。
3. **精确计量**：token 消耗从 Claude Agent SDK 的 `usage` 对象获取，利用 `cacheReadTokens` 区分管理 token 和交付 token。
4. **可复现**：固定 model 版本、temperature=0、固定 Skill 文件内容。
5. **实时可观测**：通过飞书渠道实时监控实验过程，便于 Prompt 调优。

## 二、技术栈

| 层面 | 选择 | 理由 |
|---|---|---|
| 基础框架 | disclaude（已有 TypeScript 项目） | SDK 抽象、Prompt 注入、Token 统计、错误重试已封装 |
| Agent SDK | @anthropic-ai/claude-agent-sdk | disclaude 内置的 ClaudeSDKProvider |
| 主实验模型 | GLM-5-Turbo（Anthropic 兼容接口） | 按实际 Token 挂牌价计费；通过 disclaude 的 Provider 抽象调用 `https://open.bigmodel.cn/api/anthropic` |
| Pilot 模型 | GLM-5-Turbo（同上） | 与主实验统一模型，零额外成本 |
| Robustness 模型 | DeepSeek V3（OpenAI 兼容接口） | 跨模型族验证，成本极低（~$3），可通过 disclaude 的 Provider 抽象切换 |
| PM 方法注入 | Skill 文件（SKILL.md）+ Agent 定义（.claude/agents/*.md） | 零代码修改，文件操作切换方法 |
| 状态管理 | 文件系统（Agent 写状态文件）+ TodoWrite 工具 | Agent 可持久化状态到 workspace |
| 日志 | JSON lines（disclaude 已有日志系统） | 扩展实验特定的 token/状态日志 |
| 测试执行 | Bash 工具（Agent 调用 pytest/vitest） | Agent 生成代码后自动跑测试 |
| 统计分析 | Python 脚本（scipy） | 仅用于实验后的数据分析，不涉及框架本身 |

## 三、核心架构

### 3.1 整体架构：1+3 多Agent 模式

采用「1个主Agent + 3个子Agent」的固定流水线架构。**调度模式统一**，PM方法差异完全由 Skill Prompt 驱动。

```
┌─────────────────────────────────────────────────────┐
│                  主 Agent（Pilot）                    │
│              扮演 PM/Coordinator 角色                │
│                                                     │
│  职责：                                              │
│  · 读取任务 → 创建 ATU 列表 → 写入 state.json       │
│  · 选择下一个待处理的 ATU                            │
│  · 通过 Task 工具委托子Agent执行                     │
│  · 接收子Agent返回结果 → 更新 state.json             │
│  · 按 Skill 规则决定下一步（通过/退回/下一个ATU）     │
│                                                     │
│  Skill 文件决定PM方法行为差异                         │
│  ┌─────────────────────────────────────────────┐    │
│  │ SKILL.md（5种之一）                          │    │
│  │ · 瀑布：按阶段锁死，禁止跨阶段               │    │
│  │ · Scrum：按Sprint分组ATU（3个一组）           │    │
│  │ · Kanban：持续流动，WIP≤2                     │    │
│  │ · MARS：TDD门控 + ATU局部并发采样 + 状态截断   │    │
│  │ · Evolutionary PM：TDD适应度函数 + VSR进化循环 + 遗传保留 │    │
│  │ · 无管理：自由决定顺序和粒度                  │    │
│  └─────────────────────────────────────────────┘    │
└──────────┬──────────┬──────────┬────────────────────┘
           │          │          │
     Task 工具   Task 工具   Task 工具
           │          │          │
           ▼          ▼          ▼
    ┌──────────┐ ┌──────────┐ ┌──────────┐
    │Developer │ │ Tester   │ │Reviewer  │
    │ 子Agent  │ │ 子Agent  │ │ 子Agent  │
    └──────────┘ └──────────┘ └──────────┘
         │             │             │
         └─────────────┼─────────────┘
                       ▼
              共享 workspace 文件系统
          ┌──────────────────────────────┐
          │ state.json  （主Agent独占写）  │
          │ 代码文件    （Developer写）    │
          │ review.md   （Tester写）      │
          │ review.md   （Reviewer写）    │
          └──────────────────────────────┘
```

**核心设计原则**：

1. **调度模式统一**：所有PM方法使用相同的 Dev→Test→Review 流水线。方法差异通过 Skill Prompt 注入主Agent的决策逻辑，并在必要时在 Skill 内部触发额外的仪式性消息（见下方"效度威胁与控制"）。
2. **文件系统传递上下文**：子Agent通过 Read 工具读取 state.json 和代码文件，主Agent通过 Task prompt 传递轻量指令。
3. **state.json 主Agent独占写**：模拟真实PM中「PM维护项目状态，执行者只做自己的事」。

---

**⚠️ 效度威胁与控制（重要声明）**：

统一调度架构是一个**受控的简化**，它在提升内部效度的同时也引入了三个必须承认的威胁。论文第五章与第七章"研究局限"须显式讨论：

| 效度威胁 | 原设计的问题 | 本研究的控制措施 |
|---|---|---|
| **T1：瀑布仪式被阉割** | 单 ATU 顺序流水线无法体现瀑布"阶段文档交付 + 跨阶段 review"的真实开销 | Skill 中要求瀑布主 Agent 在每个"阶段边界"生成阶段交付文档并写入 `phase-gate.md`；跨阶段返工时必须调用 Reviewer 子 Agent 重审所有该阶段 ATU |
| **T2：Scrum 仪式 token 被阉割** | "3个 ATU 一组当 Sprint"无法体现 Planning/Review/Retro 的协调开销 | Skill 中要求 Scrum 主 Agent 在每个 Sprint 边界显式输出 Planning/Review/Retro 三段独立消息（带标签），这些消息按 MOR 标注规则计入 `PLANNING` / `COORDINATION` token |
| **T3：Kanban WIP 在串行调度下形同虚设** | 主 Agent 单线程循环，WIP=1 和 WIP=2 无区别 | **升级调度器支持并行 Task 调用**：Kanban 主 Agent 可同时委派多个 Developer（由 Claude Agent SDK 的并发 Task 支持）；WIP 限制真实生效 |

**论文表述建议（第五章 5.2）**：

> 本研究采用"统一调度 + Skill 注入"的架构来控制变量，但这意味着实验测量的是**在给定调度骨架下，不同 PM 方法通过系统指令引导出的协作差异**，而非各方法在其理想实现下的差异。为缓解这一威胁，Skill 文件内置了仪式性消息生成规则（瀑布的阶段文档、Scrum 的 Planning/Review/Retro），使这些管理开销可被 token 计量；Kanban 的 WIP 限制通过并行 Task 调度真实生效。结论的外推范围限定为"在 Claude Agent SDK 流水线架构下"，不主张"在所有 Agent 框架下"。

**论文第七章"研究局限"须承认**：
- 统一调度架构对瀑布/Scrum 的仪式性开销建模是近似的，真实瀑布团队的返工成本可能被低估
- 未在多框架（AutoGen、CrewAI）上复现，无法排除"结论依赖 Claude Agent SDK 特性"的可能

### 3.2 Prompt 注入分层

```
第1层：CLAUDE.md（项目级系统指令）
  ↓
第2层：Skill SKILL.md（PM 方法行为规则）← 核心注入点
  ↓
第3层：Agent 定义 .claude/agents/*.md（工具/模型控制）
  ↓
第4层：MessageBuilder（每条消息的上下文增强）
  ↓
第5层：用户消息（任务需求 / 变更注入）
```

### 3.3 PM 方法切换机制

实验时通过 SkillManager 管理 `.claude/skills/` 目录中的文件：

```typescript
class SkillManager {
  // 切换到指定PM方法：只保留目标方法的Skill文件
  switchMethod(method: 'waterfall' | 'scrum' | 'kanban' | 'no-mgmt' | 'evolutionary'): void {
    // 1. 清空 .claude/skills/ 目录中的 PM 方法文件
    // 2. 从 skill-templates/ 复制目标方法的 SKILL.md
    // 3. SDK 在下一次查询时自动加载新的 Skill
  }
}
```

### 3.4 固定流水线调度流程

所有PM方法使用统一的调度流程，差异由 Skill Prompt 控制主Agent的决策逻辑：

```
1. 初始化阶段（主Agent / PM）
   ├── 读取任务描述（tasks/xxx/README.md）
   ├── 分析并拆分为 ATU 列表
   ├── 声明 ATU 之间的依赖关系
   └── 写入 state.json（初始状态均为 Open）

2. 执行循环（主Agent 控制，循环直到所有 ATU 为 Done）
   │
   ├── 2a. 选择下一个待处理 ATU（决策逻辑由 Skill 决定）
   │       · 瀑布：按阶段顺序，先全部需求分析，再全部设计...
   │       · Scrum：按Sprint分组（3个ATU一组），组内按依赖顺序
   │       · Kanban：按优先级+依赖拉取，WIP≤2
   │       · Evolutionary PM：按依赖拓扑序逐ATU推进，每个ATU多代进化
   │       · 无管理：自由选择
   │
   ├── 2b. Task(Developer) 委托
   │       · 主Agent在 Task prompt 中传递：「实现 ATU-XXX：{描述}。
   │         参见 state.json 获取上下文。预估 {复杂度}。」
   │       · Developer 子Agent：Read state.json → Read 依赖文件 → Write 代码
   │       · 返回：「ATU-XXX 实现完成」+ 产出文件列表
   │
   ├── 2c. 主Agent 更新 state.json（ATU → In Review）
   │
   ├── 2d. Task(Tester) 委托
   │       · 主Agent在 Task prompt 中传递：「测试 ATU-XXX，运行测试」
   │       · Tester 子Agent：Read 代码 → Run tests → Write review.md
   │       · 返回：「测试通过/失败 + 原因」
   │
   ├── 2e. 主Agent 判断测试结果
   │       · 通过 → 继续 2f
   │       · 失败 → 更新 state.json（ATU 回退到 In Progress）
   │         → Task(Developer) 修复 → 回到 2d（重试逻辑由 Skill 控制）
   │
   ├── 2f. Task(Reviewer) 委托
   │       · 主Agent在 Task prompt 中传递：「审查 ATU-XXX 的代码质量」
   │       · Reviewer 子Agent：Read 代码 → Write review.md
   │       · 返回：「审查通过/退回 + 意见」
   │
   └── 2g. 主Agent 更新 state.json
           · 通过 → ATU → Done
           · 退回 → ATU → In Progress → 回到 2b

3. 变更注入（仅 T3 任务，由框架触发）
   ├── 框架检测：在 30% ~ 70% 之间的一个随机 ATU 达到 Done 时，注入 change.md（避免被模型提前预判规律）
   ├── 主Agent 读取变更内容
   ├── 按 Skill 规则处理变更：
   │   · 瀑布：暂停当前阶段，退回需求分析阶段
   │   · Scrum：变更进入 Backlog，下个 Sprint 处理
   │   · Kanban：插入任务板，重新排序
   │   · Evolutionary PM：分析依赖图，仅对受影响ATU发起局部进化
   │   · 无管理：自由响应
   └── 继续执行循环

4. 结束（主Agent / PM）
   ├── 确认所有 ATU 为 Done
   ├── 生成最终状态报告
   └── 记录实验指标
```

**各方法在流水线上的行为差异**：

| 决策点 | 瀑布 | Scrum | Kanban | Evolutionary PM | 无管理 |
|---|---|---|---|---|---|
| ATU选择顺序 | 阶段锁死：先全部需求→全部设计→全部Dev→全部Test→全部Review | Sprint分组：3个ATU一组，组内按依赖顺序 | 按优先级拉取，In Progress ≤ WIP=2 | 按依赖拓扑序逐ATU推进，每个ATU多代进化（每代2名Developer独立实现） | 自由选择 |
| 测试失败处理 | 退回上一阶段，该阶段全部ATU重做 | 当前Sprint内修复，否则放入下个Sprint | 退回该ATU，修复后重新流转 | 遗传保留：提取最优部分解性状注入下一代，禁止丢弃全部代码 | 自行决定 |
| 审查退回处理 | 同上 | 当前Sprint内修复 | 退回该ATU | 遗传保留，重新进化 | 自行决定 |
| 变更响应 | 退回需求分析阶段，全局重来 | 进入Backlog，下个Sprint处理 | 插入任务板，重新排序 | 分析依赖图，仅对受影响ATU发起局部进化 | 自由响应 |
| Sprint/周期 | 无（阶段即周期） | Sprint=3个ATU | 无（持续流动） | 无（事件驱动+代际进化） | 无 |

### 3.6 上下文传递机制

| 通道 | 方向 | 内容 | 载体 |
|---|---|---|---|
| **Task Prompt** | 主Agent → 子Agent | 当前任务指令 + ATU描述 + 约束 | Task 工具的 prompt 参数 |
| **state.json** | 主Agent → 子Agent | 全局状态（只读） | 子Agent 通过 Read 工具读取 |
| **代码文件** | Developer → Tester/Reviewer | 实现产出 | workspace 目录中的文件 |
| **review.md** | Tester/Reviewer → 主Agent | 测试/审查意见 | 子Agent 写入，主Agent 读取后决策 |
| **Task 返回文本** | 子Agent → 主Agent | 执行结果摘要 | Task 工具的返回值 |

### 3.7 Token 统计

> **重要修订**：早期草案用 `MOR ≈ cacheReadTokens / totalTokens`，这是错误的。`cacheReadTokens` 只反映系统 prompt 缓存命中情况，在不同 PM 方法间不可比（Skill 文件长度不同），且它混杂了管理规则、角色定义、任务描述。现改为**基于消息语义标注**的定义，详见 `实验设计总表_变量控制_任务集_指标定义.md` 第 5.3.1 节。

```typescript
// disclaude SDK 提供的基础 token 统计（仅作为原始数据输入）
interface QueryUsageStats {
  inputTokens: number;
  outputTokens: number;
  totalTokens: number;
  costUsd: number;
  cacheReadTokens?: number;   // 仅记录作为审计数据，不用于 MOR
  cacheWriteTokens?: number;
}

// 每条消息同时采集语义标签
interface TaggedMessage {
  role: 'main-agent' | 'developer' | 'tester' | 'reviewer';
  content: string;
  tokens: number;
  // 消息语义标签：PLANNING / COORDINATION / REVIEW_FEEDBACK / DELIVERY / OTHER
  // 一级：规则自动标注；二级：LLM-as-judge 标注未覆盖的自然语言消息
  semanticTag: 'PLANNING' | 'COORDINATION' | 'REVIEW_FEEDBACK' | 'DELIVERY' | 'OTHER';
  taggingSource: 'rule' | 'llm-judge' | 'human-review';
}

// 实验指标计算（基于消息级聚合）
// MOR = Σ(PLANNING + COORDINATION + REVIEW_FEEDBACK) tokens / Σ(total) tokens
// MTE = Σ(DELIVERY) tokens / Σ(total) tokens
// 冗余率 = Σ(OTHER) tokens / Σ(total) tokens
// 注意：MOR + MTE + 冗余率 = 1，三者互补
```

**标注质量保障**：
- 20% 消息抽样做人工复核（研究者 + 一位同学）
- Cohen's Kappa ≥ 0.7 才认为标注可用
- Kappa < 0.7 → 迭代标签定义，重新标注
- 附录公示标签定义 + 20 条代表性样本

## 六、实验执行流程

### 阶段一：Pilot 预实验（GLM-5-Turbo，按实际 Token 挂牌价计费）
- 通过飞书群实时交互执行
- 选 1 个中等任务，跑 Evolutionary PM + Scrum + 无管理各 2 次
- 验证 Skill 文件是否稳定引导 Agent 行为
- 验证 Token 统计和指标可采集性
- 调优 Skill 内容

### 阶段二：正式实验（GLM-5-Turbo，按实际 Token 挂牌价计费）
- 通过 ExperimentRunner 批量执行
- 核心三组（Scrum / Kanban / Evolutionary PM）× 4 任务（T2×2 + T3×2）× **20 次** = 120 次
- 对照两组（瀑布 / 无管理）× 4 任务 × **5 次** = 40 次
- 总计 **280 次**
- 样本量依据见 `实验设计总表` 7.1–7.2
- 五方法联合 Kruskal-Wallis + Dunn 事后比较（Bonferroni α=0.005）

### 阶段三：Robustness Check（跨模型族，DeepSeek V3）
- 模型 A：DeepSeek V3
- 模型 B：Qwen3-Coder 或 Llama-3-70B（开源权重模型，可选）
- 方法：Evolutionary PM vs Scrum vs Kanban（3 组）
- 任务：T3-1（变更场景最敏感）
- 重复：每组每模型 5 次 = 30 次
- **目的**：验证结论不依赖 GLM 模型族，获得跨模型族证据

## 七、ExperimentRunner 设计

```typescript
// experiment/src/runner.ts

interface ExperimentConfig {
  method: 'waterfall' | 'scrum' | 'kanban' | 'no-mgmt' | 'evolutionary';
  task: string;           // 任务目录名
  run: number;            // 重复次数编号
  model?: string;         // 默认从 config 读取
  injectChange?: boolean; // 是否注入需求变更（T3任务）
}

class ExperimentRunner {
  constructor(
    private skillManager: SkillManager,
    private metricsCollector: MetricsCollector,
  ) {}

  async runExperiment(config: ExperimentConfig): Promise<ExperimentResult> {
    // 1. 切换 Skill 文件到目标方法
    this.skillManager.switchMethod(config.method);

    // 2. 读取任务输入
    const taskInput = await this.loadTask(config.task);

    // 3. 通过 disclaude CLI 或 Pilot 执行
    const result = await this.execute(taskInput, config);

    // 4. 收集指标
    const metrics = this.metricsCollector.collect(result);

    // 5. 保存结果
    await this.saveResult(config, metrics);

    return { config, metrics };
  }

  async runMatrix(): Promise<void> {
    // method × task × repeat 矩阵执行
    for (const method of METHODS) {
      for (const task of TASKS[method]) {
        for (let run = 1; run <= REPEATS; run++) {
          await this.runExperiment({ method, task, run });
        }
      }
    }
  }
}
```
