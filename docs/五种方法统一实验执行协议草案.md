# 五种方法统一实验执行协议草案

## 一、设计目的
为保证不同项目管理方法之间的比较具有可比性，本研究需要将各方法转化为统一的实验执行协议。协议的目标是：
1. 保持实验组之间的角色、工具、任务和评估规则一致；
2. 仅改变“项目管理方法”这一核心自变量；
3. 将抽象的项目管理方法落实为可执行、可记录、可复现的协作流程。

## 二、统一实验基础设置

### 2.1 统一角色设置
所有实验组均采用以下四类角色：
- PM / Coordinator：负责任务分配、进度协调与状态维护
- Developer：负责实现与修改
- Tester：负责测试执行与缺陷识别
- Reviewer：负责交付物审查与质量把关

### 2.2 统一任务输入
- 输入同一份任务需求说明
- 中等与复杂任务在预设时点注入相同变更
- 所有组使用同一批任务集

### 2.3 统一执行资源
- 相同LLM底座
- 相同上下文限制
- 相同工具权限
- 相同运行环境

### 2.4 统一记录要求
所有实验组都必须记录：
- 任务拆分结果
- 状态变化记录
- Agent通信日志
- 测试结果
- 最终交付物

## 三、五种方法的统一执行协议

### 3.1 瀑布方法执行协议
#### 核心逻辑
以顺序阶段推进为主，强调阶段完成后再进入下一阶段，每个阶段必须产出正式文档并通过审批门控。

#### 阶段定义与门控要求
| 阶段 | 必须产出的阶段文档 | 门控审批者 |
|------|-------------------|-----------|
| 需求分析 | `requirements.md` | Reviewer |
| 方案设计 | `design.md` | Reviewer |
| 开发实现 | 源代码 + `impl-notes.md` | Tester（通过测试即视为通过） |
| 测试验证 | `test-report.md` | Reviewer |
| 最终交付 | `delivery-summary.md` | Reviewer |

**门控规则**：每个阶段结束时，PM 必须提交阶段文档，Reviewer 审批通过后才能进入下一阶段。审批不通过则退回当前阶段修正。

#### 执行流程
1. PM生成需求分析结果，输出 `requirements.md`
2. Reviewer审批需求文档，通过则进入下一阶段，否则退回修正
3. PM生成系统设计，输出 `design.md`，Reviewer审批
4. PM将开发任务分配给Developer，Developer完成实现
5. Tester执行测试，输出 `test-report.md`，Reviewer审批
6. PM输出 `delivery-summary.md`，Reviewer最终验收
7. 若阶段未通过审批，则退回当前阶段修正（不可跳过阶段）

#### 管理规则
- 不允许跨阶段并行推进
- 当前阶段文档未通过审批，禁止进入下一阶段
- 需求变更优先记录，统一在阶段边界处理
- 所有阶段文档计入管理开销（MOR统计）

#### 适用假设
适合需求明确、变化较少的任务

### 3.2 Scrum方法执行协议
#### 核心逻辑
以Sprint为基本管理单元，强调固定时间盒、短周期计划/执行/评审/回顾、以及每日同步。

#### Sprint时间盒定义
- **每个 Sprint = 固定 5 轮 Agent 交互**（类比人类 Scrum 的 2 周时间盒）
- Sprint 开始时 PM 提交 Sprint Backlog 承诺
- 5 轮交互到达后，**强制进入 Sprint Review + Retrospective**，无论任务是否全部完成
- 未完成的任务放回 Product Backlog，由下一个 Sprint 重新规划

#### Daily Standup（每轮同步）
在每轮 Agent 交互开始时，各角色必须进行格式化的状态汇报：
- **Developer**：上一轮完成了什么 / 本轮打算做什么 / 是否有阻碍
- **Tester**：当前测试进展 / 发现的问题
- **Reviewer**：当前评审进展 / 需要关注的点
- **PM**：汇总状态，识别风险

实现方式：Skill 中强制要求每轮开发前输出 Standup 记录（计入 MOR 统计）。

#### 执行流程
1. PM建立Product Backlog，对所有任务进行优先级排序
2. **Sprint Planning**：PM从Backlog中选取本轮任务，形成Sprint Backlog并承诺交付
3. **Sprint Execution**（共5轮交互）：
   a. 每轮开始时执行 Daily Standup
   b. Developer执行开发任务
   c. Tester在开发完成后及时测试
   d. Reviewer在测试通过后及时评审
4. 5轮到期后强制进入 **Sprint Review**：展示本轮交付增量
5. PM组织 **Retrospective**：总结本轮经验教训，调整Backlog
6. 若有剩余任务或新增任务，则进入下一轮Sprint

#### 管理规则
- **Sprint 锁定**：Sprint Backlog 一旦确定，Sprint 执行期间**不接受新需求或变更**
- 新需求或变更统一进入 Product Backlog，由下一个 Sprint Planning 处理
- 协调在Sprint边界集中发生（Planning/Review/Retro）
- Daily Standup 提供Sprint内部的轻量同步
- 每轮Sprint必须完成最小可交付增量评审

#### 适用假设
适合可分阶段增量交付的任务

### 3.3 Kanban方法执行协议
#### 核心逻辑
以任务流转为中心，强调持续流动与在制品限制。

#### 执行流程
1. PM建立任务板并拆分任务
2. 所有任务进入初始状态（To Do）
3. Agent按规则从任务板拉取任务
4. 任务在各状态间流转：To Do → In Progress → In Review → Done
5. Tester和Reviewer在流转过程中持续介入
6. 若任务受阻，则进入Blocked状态，等待解除后重新流转

#### 管理规则
- 限制同时处于In Progress状态的任务数量
- 允许不同任务并行推进
- 需求变更可直接进入任务板并重新排优先级

#### 适用假设
适合任务粒度较小、流式处理较多的任务

---

### 3.4 无管理对照组执行协议
#### 核心逻辑
不给出明确项目管理规则，仅设置任务目标与角色。

#### 执行流程
1. 向所有Agent提供任务目标
2. Agent自主讨论、拆分与执行任务
3. Tester和Reviewer在必要时自主介入
4. 当Agent判断任务完成时提交最终结果

#### 管理规则
- 不设固定流程
- 不设任务板或Sprint机制
- 不设强制状态记录

#### 适用假设
作为最低管理约束对照组，用于检验“管理是否必要”

### 3.5 Evolutionary PM 方法执行协议
#### 核心逻辑
以 ATU（Atomic Task Unit，原子任务单元）为进化单位，以 TDD 测试作为适应度函数，通过**变异-选择-保留（VSR）循环**逐步优化交付质量。核心策略：**"用代际进化替代一次性尝试"**——每一代都在上一代最优解的基础上改进，而非独立重试。

**与 MARS 的本质区别**：MARS 在失败时丢弃全部代码重新采样（信息浪费）；Evolutionary PM 保留最优部分解作为"遗传种子"注入下一代（信息积累）。

#### ATU 状态模型（5 状态 + 代际嵌套）
- Open
- TDD Writing（编写测试用例）
- Evolving（进化中，包含多代变异-选择-保留循环）
- Done（完全通过测试）
- Extinct（达到最大代数仍未完全通过，接受最优部分解）

```
ATU 状态转移图：
Open → TDD Writing → Evolving (多代 VSR 循环) ──┬→ Done (完全通过)
                           │                      └→ Extinct (达最大代数)
                           └─ 部分通过 → 下一代继续进化

代际嵌套结构（每个 ATU 的 generations[] 数组）：
Gen N: Variation(2+ Developer) → Selection(测试评分) → Retention(性状提取+遗传)
```

#### 关键事件
- `atu_start`：ATU 开始执行（进入 TDD 测试编写阶段）
- `gen_start`：新一代进化开始
- `gen_variation`：变异阶段（委派多个 Developer 生成变体）
- `gen_selection`：选择阶段（TDD 门控测试，选出最优样本）
- `gen_retention`：保留阶段（从最优样本提取性状，注入遗传材料）
- `gen_end`：代际结束（通过 / 部分通过 / 灭绝）
- `evolution_complete`：该 ATU 通过进化成功完成
- `extinct`：达到最大代数仍未完全通过，接受最优部分解
- `atu_end`：ATU 完成

#### 执行流程
1. PM 根据需求建立 ATU 列表，初始化 `geneticMaterial` 池，写入 state.json
2. 对每个 ATU，先委派 Tester 编写针对该 ATU 的测试用例（TDD 门控——建立适应度函数）
3. **第 1 代变异**：PM 委派 2 名 Developer 独立开发，每个 Developer 在隔离分支中独立实现
4. **选择**：PM 运行测试，记录每个样本的通过测试数/总测试数，选出最优样本
5. **保留**（完全通过）：提取 2-4 条性状 → 追加到 `geneticMaterial.patterns` → ATU Done
6. **保留**（部分通过）：提取最优性状 → 记录到 `generations[].retainedTraits` → 进入第 2 代
7. **第 N 代变异**（N≥2）：PM 委派 Developer，prompt 中注入上一代性状 + `geneticMaterial.patterns` 交叉性状
8. 若达到最大代数（默认 5）仍未完全通过：接受全部代数中最优样本，标记 Extinct
9. **交叉遗传**：开始新 ATU 时，从 `geneticMaterial.patterns` 中选择 2-3 条相关性状注入第 1 代 prompt
10. 所有 ATU 完成后，生成最终状态报告

#### 管理规则
- 所有任务必须被拆分为 ATU
- 必须先写测试（TDD），测试是适应度函数
- Developer 之间相互隔离，不共享代码上下文（隔离变异环境）
- **失败时严禁丢弃全部代码**——必须保留最优部分解并提取性状（区别于 MARS 状态截断）
- 性状必须来自对实际代码的阅读（不得伪造遗传材料）
- `geneticMaterial.patterns` 跨 ATU 共享——已完成的 ATU 的性状可以注入后续 ATU
- 需求变更时，仅对受影响的 ATU 发起局部重新进化
- 事件通过 `[RECORD_EVENT]` marker 行声明，框架自动注入时间戳（禁止 Agent 自填）

#### 适用假设
适合 LLM 执行不可靠但沟通成本低的 Agent 协作场景。特别适合：任务之间存在可传递的代码模式（交叉遗传的优势场景）；单次实现的成功率不高但每次尝试都有"部分正确"的代码（渐进进化的优势场景）。

## 四、五种方法统一比较维度
| 比较维度 | 瀑布 | Scrum | Kanban | 无管理 | Evolutionary PM |
|---|---|---|---|---|---|
| 管理单元 | 阶段 | Sprint（5轮时间盒） | 任务流 | 无固定单元 | ATU（原子任务单元） |
| 控制机制 | 顺序门控 + 阶段文档审批 | Sprint节奏 + Daily Standup | 流转与WIP限制 | 自由协作 | TDD适应度函数 + VSR进化循环 + 遗传保留 |
| 沟通模式 | 阶段性文档交付 | 周期性协调 + 每轮Standup | 持续流转沟通 | 自发沟通 | 最小必要沟通 + 遗传信息传递（事件驱动） |
| 变更处理 | 阶段边界吸收 | Sprint锁定，下个Sprint处理 | 动态插入 | 无规则处理 | 局部进化（仅受影响ATU） |
| 仪式开销 | 阶段文档 + 审批 | Planning + Standup + Review + Retro | 看板状态维护 | 无 | 测试编写 + 性状提取 + 遗传材料管理 + 代际调度 |
| 预期优势 | 稳定性与可追溯性 | 迭代性与节奏感 | 灵活流动 | 低管理开销 | 进化式渐进优化，信息积累与跨 ATU 知识复用 |

## 五、实验执行建议
1. 每个任务在每种方法下至少执行3轮以上，以减少偶然性影响；
2. 每轮实验中尽量固定随机性设置（temperature=0）；
3. 复杂任务必须包含至少一种需求变更注入；
4. 结果分析时不只比较最终产出，也比较过程行为与管理开销（MOR）；
5. 对 Evolutionary PM 方法的结论应强调”渐进优化能力”和”信息累积效率”，而非绝对优越性。
