# 实验设计总表、变量控制表、任务集草案与指标定义

## 一、实验设计总表：数字化项目管理实证研究

### 1.1 实验目标
本研究实验旨在回答两个核心问题：
- Q1：传统项目管理方法（瀑布、Scrum、Kanban）对“数字劳动力”的适配性边界在哪里？（瀑布、Scrum、Kanban）在多Agent协作中的表现是否存在显著差异？
- Q2：基于”进化适应”的 Evolutionary PM 方法，是否比基于”过程控制”的传统方法具备更高的经济效率？面向AI Agent团队的进化式项目管理方法，是否在统一实验条件下表现出更高的适配性与综合效果？

### 1.2 实验总体结构

> **修订说明**：原设计将 E1（传统方法对比）与 E2（新方法 vs 最优传统方法）分开，存在 **post-hoc selection** 问题——E2 的"最优传统方法"从 E1 结果事后挑选，会放大 Evolutionary PM 优势，给评审 p-hacking 的把柄。现改为单实验多方法联合比较。

| 实验编号 | 实验名称 | 实验目的 | 对比对象 | 统计方法 |
|---|---|---|---|---|
| **E1** | **五方法联合比较实验** | 在统一条件下同时检验 5 种方法的差异 + Evolutionary PM 与各传统方法的两两对比 | 瀑布 / Scrum / Kanban / Evolutionary PM / 无管理（五组并列） | Friedman 总体显著性检验（针对配对样本） + 事后 Dunn 检验 + Bonferroni 校正 |
| E2 | 需求变更适应性子实验 | 在 T3 任务上专门采集 RWR / CRT / RWR-Token 指标 | 全部方法（E1 中 T3 任务的数据子集） | 同上 |
| E3 | 任务难度稳健性分析 | 分析方法在 T1/T2/T3 上的表现边界 | 全部方法（E1 中按任务难度切分） | 描述性统计 + 分层分析 |
| E4 | Robustness Check | 在第二个模型族上复现核心结论 | Evolutionary PM vs Scrum vs Kanban（三组） | 同 E1，小规模 |

**关键原则**：所有假设检验**只在 E1 中做一次**，事后比较做 Bonferroni 校正（5 组共 C(5,2)=10 对比较，α = 0.05/10 = 0.005）。不再做"先挑最优再对比"的二次检验。

## 二、实验对象与对比组定义表

### 2.1 对比组设置
| 组别 | 方法名称 | 管理逻辑 | 主要特征 |
|---|---|---|---|
| G1 | 瀑布 | 顺序阶段推进 | 强计划、强阶段门、低灵活性 |
| G2 | Scrum | 迭代式管理 | Sprint驱动、强调计划/评审/回顾 |
| G3 | Kanban | 流式任务管理 | 可视化流转、WIP限制、拉式分配 |
| G4 | 无管理 | 自由协作 | 不设明确流程与控制规则 |
| G5 | Evolutionary PM（新方法） | 变异-选择-保留循环+遗传优化 | 利用LLM非确定性，通过多代进化和性状遗传逐步优化交付质量 |

### 2.2 各方法在实验中的实现原则
#### 瀑布组
- 按需求分析、设计、实现、测试、交付顺序推进
- 前一阶段完成后才能进入下一阶段
- 允许阶段评审，但不鼓励中途返工

#### Scrum组
- 任务拆分为 backlog
- 按 sprint 执行
- 每轮包含 planning、review、retrospective
- 允许每轮迭代后调整 backlog

#### Kanban组
- 所有任务进入统一任务板
- Agent 按状态流转领取任务
- 设置在制品限制（WIP）
- 持续流动推进，不设 sprint

#### 无管理组
- 只给定目标和角色
- 不提供明确流程控制
- Agent 自主协作完成任务

#### 新方法组（Evolutionary PM）
- 以 ATU（Atomic Task Unit）为进化单位
- 坚持测试驱动开发（TDD）作为适应度函数
- 每个 ATU 通过**多代进化**逐步优化：每代委派 2 名 Developer 独立实现（变异），TDD 测试评分（选择），提取最优性状并遗传给下一代（保留）
- 若一代未完全通过，保留最优部分解的性状作为遗传种子注入下一代（而非丢弃重来）
- 已完成 ATU 的性状通过交叉遗传机制传递给后续 ATU
- 若达到最大代数（默认 5）仍未完全通过，接受最优部分解（灭绝）

## 四、任务集设计草案

### 4.1 简单任务
**任务特点**
- 单目标
- 单模块或低耦合
- 需求清晰稳定
- 无中途变更

**示例**
- 开发命令行待办工具
- 实现计算器 API
- 完成单页面表单校验功能

**实验价值**
- 检验管理方法在低复杂度任务下是否存在“过度管理”问题
- 比较各方法的基础开销

### 4.2 中等任务
**任务特点**
- 多模块协作
- 存在测试与文档要求
- 需要一定程度任务分工

**示例**
- 开发带前后端的小型博客系统
- 开发支持用户登录和数据管理的 Web 工具
- 完成一个含接口、页面、测试、说明文档的小型项目

**实验价值**
- 检验方法对正常复杂度协作任务的组织能力
- 比较不同方法的质量-效率平衡

### 4.3 复杂任务
**任务特点**
- 多模块联动
- 需求存在模糊性
- 中途可能插入约束变化或新需求
- 更容易出现返工和阻塞

**示例**
- 带权限管理和统计模块的任务管理系统
- 含多个服务组件的小型应用
- 需求不完全明确的软件原型项目

**实验价值**
- 检验方法在高复杂度场景下的稳定性
- 检验项目控制和重规划能力

### 4.4 需求变更注入设计
| 变更类型 | 示例 | 观察重点 |
|---|---|---|
| 功能增加 | 新增导出/筛选/统计功能 | 局部增量吸收能力 |
| 约束变化 | 改成本地存储、限制依赖库 | 重规划效率 |
| 质量要求变化 | 增加测试覆盖率/文档要求 | 返工成本与质量保持 |
| 优先级变化 | 某功能提前交付 | 调度与协调能力 |

| 指标 | 含义 | 获取方式 |

| 文档完整性 | 是否提供必要说明与交付文档 | 文档检查表 |
| 最终交付完成度 | 是否完成核心功能与必要组件 | 任务验收标准 |

### 5.2 过程效率与成本指标 (Process Efficiency & Cost)
| 指标 | 含义 | 获取方式 |
|---|---|---|
| 项目名义资源成本 (Resource Cost, 换算为 CNY) | 协作过程中的总资源消耗 | 日志统计 |
| 交互轮次数 | Agent之间消息轮次 | 对话日志统计 |
| 交付周期 (Lead Time) | 从开始到交付的总耗时 | 系统记录 |
| 冗余沟通比例 | 无增量价值的沟通占比 | 对话编码分析 |

### 5.3 管理协作损耗指标 (Coordination Loss)
| 指标 | 含义 | 获取方式 |
|---|---|---|
| 状态可追踪性 | 项目进展是否清晰可见 | 状态记录完整度 |
| 阻塞识别及时性 | 是否能快速发现并报告阻塞 | 阻塞发生到识别的时间 |
| 变更响应效率 | 面对需求变更的反应速度 | 变更注入到重规划完成的时间 |
| 无效协作损耗比 (Process Waste Ratio, PWR/MOR) | 协调/计划/评审反馈类消息 token 占总 token 比例 | 消息级语义标注 + token 加权（定义见 5.3.1） |
| 管理 Token 效率（MTE） | 交付物类消息 token 占总 token 比例 | `1 - MOR`（定义见 5.3.1） |
| 返工率 | 已完成任务因变更或错误被重新打开的比例 | 状态流回退统计 |
| 采样效率 | 冗余采样带来的质量提升与 token 开销之比 | `(最优样本测试通过率 - 首次通过率) / 多余采样token` |

#### 5.3.1 MOR / MTE 的操作化定义（重要）

> **修订说明**：早期草案曾将 `MOR ≈ cacheReadTokens / totalTokens`。该定义有严重缺陷：`cacheReadTokens` 只反映系统 prompt 的缓存命中情况，其绝对量受 Skill 文件长度影响，在不同 PM 方法之间不可比。**现改为基于消息语义标注的定义**。

**标签体系**：实验过程中每一条 Agent 消息（包括主 Agent 决策输出、Task prompt、子 Agent 返回）按以下 5 类打标签：

| 标签 | 定义 | 示例 |
|---|---|---|
| `PLANNING` | ATU 拆分、依赖声明、Sprint 规划、阶段规划等前置计划活动 | 「将任务拆分为 ATU-001 至 ATU-007，依赖关系如下…」 |
| `COORDINATION` | 状态更新、任务委派指令、WIP 控制、事件响应决策 | 「ATU-002 状态 Open → In Progress，委派 Developer」 |
| `REVIEW_FEEDBACK` | 测试意见、代码评审意见、退回原因 | 「测试失败：边界条件未处理，退回 In Progress」 |
| `DELIVERY` | 实际交付物：代码、测试代码、文档 | 代码文件内容、pytest 用例 |
| `OTHER` | 闲聊、自言自语、无信息增量内容 | 冗余重复、模型幻觉输出 |

**防废话污染（防失真）机制**：
为避免 LLM 表达风格啰嗦或生成大量解释性文本导致指标失真，在统计 MTE 和 MOR 的 Token 时，必须剔除 Markdown 代码块（```...```）外的闲聊文本，或引入「有效信息密度」加权。仅将真正体现了"管理意图"的动态生成文本算作管理开销。\n- **MOR 归一化**：在分母中扣除静态 Skill/System Prompt Token（static_tokens），确保指标不受指令文件长度干扰。

**计算公式**：

```
MOR = (PLANNING_tokens + COORDINATION_tokens + REVIEW_FEEDBACK_tokens) / (total_tokens - static_tokens)
MTE = DELIVERY_tokens / total_tokens
冗余率 = OTHER_tokens / total_tokens
注：MOR + MTE + 冗余率 = 100%（三者互补，非 1 - MOR 关系）
```

**标注流程**：

1. **一级标注**：基于规则的自动标注脚本——
   - 主 Agent 写 `state.json` 的输出 → `COORDINATION`
   - Task prompt 的 system 部分 + 委派指令 → `COORDINATION`
   - Developer 子 Agent 的 Write 工具输出到代码文件 → `DELIVERY`
   - Tester/Reviewer 写 `review.md` → `REVIEW_FEEDBACK`
   - 主 Agent 初始化阶段的 ATU 列表生成 → `PLANNING`

2. **二级标注**：规则覆盖不了的自然语言消息，用 **LLM-as-judge**（GPT-4o-mini 或 Claude Haiku）按上述 5 类打标。

3. **一致性验证**：
   - 抽样 20% 消息做**人工复核**（研究者本人 + 一位同学），计算 Cohen's Kappa ≥ 0.7 作为标注质量门槛。
   - Kappa < 0.7 时迭代优化标签定义，重新标注。

**论文中的呈现**：
- 附录给出完整标签定义 + 20 条代表性标注样本
- 正文报告 Kappa 值和标注者分歧案例
- MOR/MTE 结果报告为「均值 ± 标准差」+ 非参数检验

**为何这套定义可立住**：
- **方法间可比**：每种 PM 方法产生的消息都用同一套规则打标，不受 Skill 长度差异影响
- **可复现**：标签规则明确，LLM-as-judge 可重跑
- **可验证**：Kappa 是硬指标，评审可直接质询

#### 5.3.2 框架级代理指标：Agent-Origin MOR（`MOR_AO`）

> **实装背景**：完整的消息级 MOR 标注依赖 LLM-as-judge + 人工 Kappa 校验，**采集成本高**。为让主实验在运行时即可输出一个可快速观察的代理指标，实验框架（`experiment/src/metrics-collector.ts`）额外实装了一个基于"消息来源"的自动化代理指标 `MOR_AO`（Agent-Origin MOR）。

**定义**：

```
MOR_AO = subAgentTokens / (mainAgentTokens + subAgentTokens)
```

其中：
- `mainAgentTokens`：主 Agent（协调层）的 input + output token 总和
- `subAgentTokens`：通过 Task 工具委派给子 Agent（Developer / Tester / Reviewer）的 input + output token 总和
- 按消息的 `parentToolUseID` / `agentId` 字段自动分桶，无需人工标注

**与权威 MOR 的关系**：

| 维度 | 权威 MOR（5.3.1） | 代理 MOR_AO（5.3.2） |
|---|---|---|
| 粒度 | 消息级 × 5 类语义标签 | 消息级 × 2 类来源（main / sub） |
| 采集 | 规则 + LLM-as-judge + 20% 人工复核 | 框架运行时自动分桶 |
| 可得性 | 实验后批量标注 | 实验即刻可得 |
| 方法间可比 | 高（标签定义统一） | 中（依赖各方法是否真实使用子 Agent 委派） |
| 在论文中的角色 | **主指标** | **辅助指标 / 健康度监控 / Pilot 阶段快速判读** |

**互补使用策略**：
- `MOR_AO` 高 → 说明该方法**实际发生了**角色委派协作（Scrum/Evolutionary PM 应表现为高）
- `MOR_AO ≈ 0` → 说明主 Agent 一个人做完全部工作（"无管理"组应表现为接近 0）
- 若某方法宣称有多角色协作但 `MOR_AO` 却接近 0 → 说明 **Skill Prompt 没激活角色分派**，需要先修 Prompt 再做正式实验
- 正式论文**仍以 5.3.1 的权威 MOR 为主分析**；`MOR_AO` 用于：(a) Pilot 阶段方法实现正确性验证；(b) 正文附录作为独立证据支持

**实装状态**：
- 已在 `metrics-collector.ts` 实装 `mainAgentTokens / subAgentTokens / byAgent` 三个字段
- 已在 `agent-runner.ts` 透传消息的 `parentToolUseID` / `agentId`
- 单元测试覆盖（见 `tests/metrics-collector.test.ts`）
- `result.json` 每次运行自动输出

### 5.4 需求变更适应性指标（T3任务专用）

> 以下三个指标专门用于评估不同PM方法在需求变更场景下的局部重规划能力。仅在T3（复杂任务+需求变更）实验中采集。

| 指标 | 缩写 | 定义 | 计算公式 | 获取方式 | 预期差异 |
|---|---|---|---|---|---|
| **变更响应返工范围** | RWR | 需求变更后，状态发生回退的ATU/Issue数量占总量的比例 | `回退的Issue数 / 总Issue数 × 100%` | 变更注入后的状态转移日志 | Evolutionary PM < Kanban < Scrum < 瀑布 |
| **变更响应时间** | CRT | 从需求变更注入到所有受影响Issue重新达到Done的耗时 | `最后一个受影响Issue变为Done的时间 - 变更注入时间` | 事件日志时间戳 | Evolutionary PM < Kanban < Scrum < 瀑布 |
| **返工Token浪费率** | RWR-Token | 因变更导致重做已完成工作所消耗的token占变更前已完成工作token的比例 | `变更后重做消耗的token / 变更前已完成工作的token × 100%` | 变更前后的token日志对比 | Evolutionary PM最低 |

**指标设计说明**：

1. **RWR（变更响应返工范围）**：直接回答”变更发生后，有多少工作被浪费了？”。Evolutionary PM的局部进化机制应使返工范围最小——仅对受影响ATU发起局部进化；瀑布可能退回设计阶段导致全局返工；Scrum将变更放入下一Sprint导致延迟响应。

2. **CRT（变更响应时间）**：直接回答”变更发生后，多久能恢复交付？”。这衡量的是方法的响应速度而非最终质量。

3. **RWR-Token（返工Token浪费率）**：直接回答”变更导致浪费了多少token？”。这是Token-Aware维度的核心证据，传统PM理论从未涉及。

**论文中的使用建议**：
- RWR、CRT、RWR-Token 三个指标联合使用，从”范围-时间-成本”三个角度全面刻画变更适应性
- 仅在T3任务中采集，T1/T2不涉及需求变更
- 在第六章实验结果中作为独立小节分析，支撑RQ4的结论

### 5.5 综合评价思路
- 质量维度反映”做得对不对”
- 效率维度反映”做得省不省”
- 管理适配性维度反映”管得合不合适”
- 变更适应性维度反映”变起来快不快、省不省”（T3专用）
- 可采用加权综合评分，也可分维度独立比较，避免单一总分掩盖方法差异

## 六、实验平台与技术方案

> 原框架选型为 LangGraph，经决策调整为 **Claude Agent SDK (TypeScript/Node.js)** 自研轻量框架。详见 `experiments/实验框架技术方案.md`。

### 6.1 技术平台（已实装）

**框架代码**：`experiment/` 目录，基于 `@anthropic-ai/claude-agent-sdk` 自研，TypeScript + ESM。开源于 https://github.com/Harr0y/mem-agent-pm。

**8 模块单一职责架构**：
| 模块 | 职责 |
|---|---|
| `config-loader` | 读 `.env` → 运行时配置（apiKey / baseUrl / model） |

| `agent-runner` | 包装 SDK.query()，流式收集消息，按来源分桶 token |
| `test-runner` | 执行 pytest，解析通过率，检测工作区布局异常 |
| `metrics-collector` | 汇总 token / 消息数 / 时长，按 agent 来源分桶（见 5.3.2） |
| `event-recorder` | 框架侧注入 ISO-8601 时间戳 + schema 校验的事件流 |
| `experiment-runner` | 顶层编排，串联上述模块 |

**开发纪律**：
- TDD 开发：72 个 vitest 单元测试，全绿
- 方法论实现位于 `experiment/skill-templates/<method>/`，**五种 PM 方法的差异仅体现在 skill 内容中**，框架代码不为特定方法做分支
- **设计目的**：论证 Evolutionary PM 作为方法论的框架无关性——自研轻量框架即可实现全部 PM 方法，无需依赖特定编排工具

**可复现性保障**：
- 每次运行输出 `result.json`（summary）+ `messages.jsonl`（完整消息流）+ `experiment.jsonl`（事件流）+ `workspace/state.json`（Agent 状态）
- `result.json` 结构在 v0.2 后仅保留摘要（12 KB/run），完整消息流落入 `messages.jsonl`，便于 160+ runs 批量归档
- 事件时间戳由框架侧 monotonic clock 注入（`event-recorder.ts`），不依赖 Agent 自报，排除时间相关指标（如 CRT）的伪造空间

**数据有效性修复履历**（2026-04-19 已完成 9 项修复）：
- MOR token 按 agent 来源分桶（见 5.3.2）
- 事件时间戳由框架注入 ISO-8601，禁止 Agent 自填
- pytest 布局守卫（rogue .py 检测 + CLAUDE.md 强制代码位置）
- result.json 瘦身（-99%）

## 七、实验规模与统计分析

> 以下内容整合自原「实验规模、轮次、统计检验与成本估算草案」。

### 7.1 分阶段实验策略

> **修订说明**：原每组 5 次重复的样本量对 LLM 高方差输出几乎注定测不出显著性（Mann-Whitney U 对 n=5 vs n=5 最小可达 p=0.008，要求效应量极大）。核心对比现上调至 n=10。

#### 阶段一：Pilot 预实验
- 方法：Scrum、Kanban、Evolutionary PM（3组）
- 任务：中等任务 1 个、复杂任务 1 个
- 重复：每组每任务 3 次 → 共 18 次
- 模型：GLM-5-Turbo（按实际 Token 挂牌价计费）
- **目的**：校准 Skill Prompt 稳定性、MOR 标注规则、指标可采集性

#### 阶段二：正式五方法联合比较（E1 主体实验）
- 方法：瀑布、Scrum、Kanban、Evolutionary PM、无管理（5 组）
- 任务：简单 1 个 (T1-1)、中等 2 个 (T2-1, T2-2)、复杂 1 个 (T3-1)
- 模型：GLM-5-Turbo（按实际 Token 挂牌价计费）
- 重复：
  - **Evolutionary PM、Scrum、Kanban**（核心对比三组）：每组每任务 **20 次** → 240 次
  - **瀑布、无管理**（对照组）：每组每任务 **5 次** → 40 次（维持不变以控制成本）
- 合计：**280 次**（包括对照组扩样）
- **目的**：回答 RQ1–RQ4 主假设

#### 阶段三：Robustness Check（E4）
- 模型：DeepSeek V3 + 一个开源模型（Qwen3-Coder 或 Llama-3-70B，可选）
- 方法：Evolutionary PM vs Scrum vs Kanban（3 组）
- 任务：T3-1（变更场景最敏感）
- 重复：每组每模型 5 次 → 30 次
- **目的**：验证结论不依赖 GLM 模型族，获得跨模型族证据

**样本量依据**：按 Mann-Whitney U 双侧检验、中等效应量（r=0.5）、α=0.005（Bonferroni 校正后）、power=0.8 粗估，核心三组需 n≥9。考虑到 LLM 方差极大，取 n=20 留安全余量以捕获长尾分布。

### 7.2 样本量决策依据

| 对比类型 | 样本量 | 期望效应量 | 备注 |
|---|---|---|---|
| 核心 H1：Evolutionary PM vs Scrum（效率） | n=20 | r ≥ 0.5（中等-大） | 最关键对比，不能缩 |
| 核心 H2：Evolutionary PM vs Kanban（变更适应性） | n=20 | r ≥ 0.5 | T3 数据子集 |
| 边界对照：瀑布/无管理 | n=5 | — | 只用于描述性证据与边界说明 |
| 五方法总检验（Friedman） | 合计 ≥ 40 | η² ≥ 0.15 | 自由度 4，总样本越大越稳 |

### 7.3 随机性控制
- temperature 固定为 0（或模型允许的最小值）
- 固定系统 Prompt、任务输入、变更注入时点（见 E2 的时点控制）
- 每次实验使用独立 workspace，避免状态污染
- **LLM 非确定性说明**：即便 temperature=0，工具调用顺序与 cache 命中也可能导致轻微差异，论文中需显式承认

### 7.4 统计分析流程

1. **描述性统计**：每组报告 n、均值、标准差、中位数、IQR、最小/最大值

2. **主检验（E1 一次性做完，不做二次挑选）**：
   - **Friedman**：五组总检验，判断是否存在组间差异
   - 若显著 → **Nemenyi 事后检验（或 Wilcoxon 符号秩检验 + Bonferroni 校正）**，共 C(5,2)=10 对
   - **Bonferroni 校正**：α_调整 = 0.05/10 = 0.005

3. **效应量报告**（统计显著≠实际显著）：
   - 两两比较报告 **Cliff's delta** 或 **rank-biserial correlation r**
   - r < 0.3 小效应 / 0.3–0.5 中等 / ≥ 0.5 大效应
   - **即便 p 未显著，只要 r ≥ 0.5 且方向一致，仍可在讨论章节报告为"具有实践意义的差异"**

4. **可视化**：箱线图（分布）+ 小提琴图（密度）+ 折线图（变更前后变化）

5. **预注册声明**（建议）：在 pilot 阶段结束后、正式实验前，将假设、样本量、指标定义、分析方法写入论文附录并做时间戳，防止结果出来后调整分析方法被质疑。

- **可缩减**：简单任务数量、瀑布重复次数、无管理组轮次
